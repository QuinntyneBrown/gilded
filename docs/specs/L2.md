# L2 - Detailed Requirements

## L2-001: User Registration
**Traces to:** L1-009

Users must be able to create an account using email and password, with email verification before account activation.

**Acceptance Criteria:**
1. Given a visitor on the signup page, when they submit a valid email and a password meeting complexity rules (min 12 chars, mixed case, number, symbol), then an account is created in `pending_verification` state and a verification email is sent.
2. Given a visitor submits an email already registered, when they submit, then the system responds with a generic "check your email" message without disclosing account existence.
3. Given a user clicks a valid verification link within 24 hours, when the link is processed, then the account is activated.
4. Given a verification link older than 24 hours, when clicked, then the system rejects it and offers to resend.

---

## L2-002: User Login and Session Management
**Traces to:** L1-009

Users must be able to log in with email and password and receive a secure session.

**Acceptance Criteria:**
1. Given valid credentials on an active account, when the user logs in, then a session is created with an HTTP-only, Secure, SameSite=Lax cookie and a session lifetime of 30 days with sliding expiration.
2. Given invalid credentials, when the user attempts login, then the system returns a generic error and increments a rate-limit counter.
3. Given 5 failed attempts within 15 minutes for the same account or IP, when another attempt is made, then the system rejects it with HTTP 429.
4. Given a logged-in user, when they click logout, then the session is invalidated server-side.

---

## L2-003: Password Reset
**Traces to:** L1-009

Users must be able to reset forgotten passwords via email.

**Acceptance Criteria:**
1. Given a user requests a reset for any email, when submitted, then the system always responds success (no account enumeration) and, if the account exists, sends a single-use token valid for 1 hour.
2. Given a valid reset token, when the user submits a new compliant password, then the password is updated, all existing sessions are invalidated, and the token is consumed.
3. Given an expired or reused token, when submitted, then the system rejects it.

---

## L2-004: Spouse Linking
**Traces to:** L1-001

Two user accounts must be able to be linked as spouses to form a couple.

**Acceptance Criteria:**
1. Given a logged-in user, when they send a spouse-link invitation to an email, then an invitation token valid for 7 days is emailed to that address.
2. Given the invited user accepts while logged in (or after signup), when accepted, then both accounts are joined to a single `couple` record and each user's `spouseId` is set.
3. Given a user already in a couple, when they attempt to send or accept another invitation, then the system rejects it with a clear error.
4. Given a user in a couple, when they choose "unlink spouse", then the couple is dissolved, spouse-shared notes become private to their original authors, and both users may form new couples.

---

## L2-005: Postal Code Proximity Search
**Traces to:** L1-002

Users must be able to search counsellors by postal code and radius.

**Acceptance Criteria:**
1. Given a user enters a valid Canadian or US postal/ZIP code and a radius (default 25 km), when they search, then results are returned ordered by ascending distance from the postal centroid.
2. Given an invalid postal code format, when submitted, then the system returns a validation error without calling the geocoder.
3. Given a search that returns more than 20 results, when rendered, then results are paginated at 20 per page.
4. Given a search, when it executes, then the p95 response time must be <= 1000 ms for cached postal codes and <= 2500 ms for uncached.

---

## L2-006: Counsellor Profile
**Traces to:** L1-002, L1-003

Each counsellor must have a profile with identifying, contact, and faith-relevant information.

**Acceptance Criteria:**
1. Given a counsellor record, when viewed, then the profile displays: name, denomination/faith tradition, credentials, specialties, address, phone, email, website, booking link, and source (`web_research` or `user_submitted`).
2. Given a counsellor profile, when displayed, then aggregate rating (1-5 stars, one decimal), review count, and distance from the viewer's saved postal code (if set) are shown.
3. Given a counsellor with no reviews, when displayed, then the rating area shows "No reviews yet" rather than 0 stars.

---

## L2-007: Web Research Ingestion Pipeline
**Traces to:** L1-003

The system must ingest counsellor records from an automated web research process.

**Acceptance Criteria:**
1. Given the ingestion job runs, when a new counsellor is discovered, then the record is persisted with `source=web_research`, `verified=false`, and a `sourceUrl` pointing to the origin.
2. Given an ingested counsellor matches an existing record by (name + address) or (phone) or (email), when processed, then the record is merged rather than duplicated.
3. Given ingestion encounters a record missing required fields (name, location), when processed, then the record is rejected and logged.
4. Given the ingestion job, when it runs, then it must not expose third-party API keys in logs or error messages.

---

## L2-008: Manual Counsellor Submission
**Traces to:** L1-004

Authenticated users must be able to add a counsellor that does not already exist.

**Acceptance Criteria:**
1. Given a logged-in user on the "Add Counsellor" form, when they submit required fields (name, address or postal code, at least one contact method, denomination), then a record is created with `source=user_submitted`, `verified=false`, and `submittedBy=<userId>`.
2. Given a submission matches an existing counsellor by (name + address) or (phone) or (email), when submitted, then the system surfaces the existing record and blocks duplicate creation.
3. Given a submission, when saved, then the system queues a moderation review before the record becomes publicly searchable.
4. Given a moderator rejects a submission, when rejected, then the submitter is notified with a reason.

---

## L2-009: Rating a Counsellor
**Traces to:** L1-005

Users must be able to rate a counsellor on a 1-5 star scale.

**Acceptance Criteria:**
1. Given a logged-in user on a counsellor profile, when they submit a rating between 1 and 5 (integers), then the rating is stored keyed by (userId, counsellorId) and the aggregate is recomputed.
2. Given a user has already rated a counsellor, when they submit a new rating, then the prior rating is updated (one rating per user per counsellor).
3. Given a rating outside 1-5, when submitted, then the system rejects it with HTTP 400.

---

## L2-010: Reviews and Comments
**Traces to:** L1-005

Users must be able to write text reviews and comment on reviews.

**Acceptance Criteria:**
1. Given a logged-in user, when they post a review of 20-4000 characters, then it is stored with author, counsellorId, timestamp, and visibility=public.
2. Given any user, when they post a comment of 1-1000 characters on a review, then it is stored with author, reviewId, and timestamp.
3. Given a review or comment containing disallowed content (per moderation ruleset), when submitted, then it is rejected or flagged per policy.
4. Given the author of a review or comment, when they delete it, then the content is soft-deleted and replaced with "[removed by author]".
5. Given a moderator, when they remove content, then it is replaced with "[removed by moderator]".

---

## L2-011: Counsellor Shortlist and Evaluation
**Traces to:** L1-006

A couple must be able to build a shortlist and compare candidate counsellors.

**Acceptance Criteria:**
1. Given a user in a couple, when they click "Shortlist" on a counsellor, then that counsellor is added to the couple's shared shortlist.
2. Given a couple's shortlist with 2+ counsellors, when the user opens the comparison view, then a side-by-side grid displays rating, distance, denomination, specialties, and price/insurance fields.
3. Given a user in a couple, when either spouse marks a counsellor as "Chosen", then the couple's `chosenCounsellorId` is set and the other spouse is notified.

---

## L2-012: Appointment Intent
**Traces to:** L1-007

A couple must be able to record the intent to make an appointment with the chosen counsellor.

**Acceptance Criteria:**
1. Given a couple with a chosen counsellor, when a user clicks "Make Appointment", then the counsellor's booking link, phone, and email are displayed, and an `appointment_intent` event is recorded (couple, counsellor, timestamp).
2. Given the counsellor has a booking link, when clicked, then it opens in a new tab with `rel="noopener noreferrer"`.
3. Given an appointment intent exists, when the couple revisits the app, then a banner reminds them of the pending appointment and prompts them to mark it "Booked" or "Cancelled".

---

## L2-013: Private Notes
**Traces to:** L1-008, L1-010

Users must be able to write notes visible only to themselves.

**Acceptance Criteria:**
1. Given a logged-in user, when they create a note with `visibility=private`, then it is stored and retrievable only by the author.
2. Given any user other than the author (including the spouse and admins), when they query notes by any API, then private notes are never returned.
3. Given a private note, when stored, then its content is encrypted at rest with a key not shared across tenants.

---

## L2-014: Spouse-Shared Notes
**Traces to:** L1-008, L1-010

Users must be able to write notes visible only to themselves and their linked spouse.

**Acceptance Criteria:**
1. Given a user in a couple, when they create a note with `visibility=spouse`, then it is stored and retrievable only by the author and their linked spouse.
2. Given a user not in a couple, when they attempt to create a spouse-shared note, then the system rejects the request with a clear error.
3. Given a couple is dissolved (L2-004 AC4), when a spouse queries spouse-shared notes, then only notes they authored are returned; notes authored by the former spouse are no longer accessible.
4. Given a spouse-shared note, when stored, then its content is encrypted at rest.

---

## L2-015: Public Notes
**Traces to:** L1-008

Users must be able to write notes visible to all platform users.

**Acceptance Criteria:**
1. Given a logged-in user, when they create a note with `visibility=public`, then it is stored and readable by all authenticated users.
2. Given a public note is created, when submitted, then it passes the same moderation ruleset as reviews (L2-010 AC3).
3. Given a public note, when displayed, then the author's display name (not email) is shown.

---

## L2-016: Note Authorization Enforcement
**Traces to:** L1-010

Note visibility must be enforced at the data-access layer, not only in the UI.

**Acceptance Criteria:**
1. Given any note-fetching API endpoint, when called, then it filters results based on the caller's identity and couple relationship before returning.
2. Given a user tampers with a note ID in a direct URL or API call, when the request is made, then access is denied with HTTP 403 if the user is not authorized.
3. Given automated tests for IDOR (insecure direct object reference), when run against note endpoints, then all tests must pass.

---

## L2-017: Mobile-First Responsive Layouts
**Traces to:** L1-011

All primary screens must render correctly across mobile, tablet, and desktop breakpoints.

**Acceptance Criteria:**
1. Given viewport width < 576px, when the user views any primary screen (search, profile, shortlist, notes), then the layout is single-column with tap targets >= 44x44px.
2. Given viewport widths at 576, 768, 992, and 1200px, when the user views primary screens, then no horizontal scroll appears and no content is clipped.
3. Given a mobile viewport, when the user interacts with forms, then inputs use appropriate mobile keyboard types (email, tel, numeric).

---

## L2-018: Search and Page Performance
**Traces to:** L1-012

Core flows must meet response-time budgets.

**Acceptance Criteria:**
1. Given a cold load on a mid-tier mobile device over 4G, when any primary page loads, then Largest Contentful Paint must be <= 2.5s.
2. Given authenticated API calls for notes list, shortlist, or profile view, when executed at p95, then server response time must be <= 500 ms.
3. Given 100 concurrent users performing searches, when the system is under load, then error rate must be < 0.5%.

---

## L2-019: Input Validation and Injection Prevention
**Traces to:** L1-013

All user input must be validated and safely handled.

**Acceptance Criteria:**
1. Given any form submission, when processed, then inputs are validated by a server-side schema before persistence.
2. Given user-generated content is rendered, when output, then it is HTML-escaped or sanitized against a strict allowlist.
3. Given any database query, when executed, then it uses parameterized statements; no string concatenation of user input into queries is permitted.
4. Given file or image uploads (e.g., counsellor photos), when received, then MIME type, size (<= 5 MB), and dimensions are validated and the file is stored outside the web root.

---

## L2-020: Rate Limiting and Abuse Controls
**Traces to:** L1-013

The system must rate-limit abuse-prone endpoints.

**Acceptance Criteria:**
1. Given login, signup, password reset, and spouse-invite endpoints, when called more than 10 times per IP per minute, then subsequent calls return HTTP 429.
2. Given review, comment, and public-note creation, when a single user creates more than 20 items per hour, then further creations return HTTP 429.
3. Given suspected automated abuse, when detected, then a CAPTCHA challenge is presented on signup and review creation.

---

## L2-021: Logging and Observability
**Traces to:** L1-014

The system must log and emit metrics for operational visibility while protecting confidential content.

**Acceptance Criteria:**
1. Given any request, when handled, then a structured log entry with request ID, user ID (if present), route, status, and latency is emitted.
2. Given any log entry, when emitted, then note content, passwords, tokens, and full email addresses are never included.
3. Given the system is running, when metrics are scraped, then request rate, error rate, p50/p95/p99 latency, and authentication failure rate are available.

---

## L2-022: Accessibility
**Traces to:** L1-015

All screens must meet WCAG 2.1 AA.

**Acceptance Criteria:**
1. Given any screen, when audited by an automated tool (axe-core), then zero critical or serious violations are reported.
2. Given any interactive element, when navigated by keyboard, then focus order is logical and focus is visible.
3. Given any image, icon-only button, or form input, when rendered, then it has an accessible name (alt text, aria-label, or associated label).
4. Given any text or UI element, when rendered, then color contrast meets WCAG AA ratios (4.5:1 body, 3:1 large text/UI).

---

## L2-024: Angular Material Exclusivity
**Traces to:** L1-016

The entire frontend must use Angular Material components, theming, and design tokens. Custom or third-party UI libraries, hand-rolled component styling, and ad-hoc color/typography values are prohibited.

**Acceptance Criteria:**
1. Given the application is built with Angular (v17+) and `@angular/material` is installed, when any interactive UI element is rendered (button, input, select, checkbox, radio, slider, toggle, menu, tabs, stepper, chip, tooltip, progress indicator, table, paginator, sort header, expansion panel, card, list, tree, badge, snackbar), then it must be the corresponding Angular Material component (e.g., `<button mat-button>`, `<mat-form-field>`, `<mat-select>`, etc.) rather than a raw HTML element styled manually or a non-Material library.
2. Given any dialog, modal, bottom sheet, or confirmation prompt is shown, when opened, then it must be implemented via `MatDialog` or `MatBottomSheet`.
3. Given any top-level navigation, side navigation, or app toolbar is rendered, when displayed, then it must use `MatToolbar` and `MatSidenav`/`MatNavList` (or `MatTabNav` where appropriate).
4. Given any text is rendered, when styled, then typography must come from the Angular Material typography system via the `mat.typography-hierarchy` mixin and `mat-typography` classes (e.g., `mat-headline-1`, `mat-body-1`); custom `font-family`, `font-size`, `font-weight`, or `line-height` declarations outside the Material typography config are prohibited.
5. Given any color is applied (background, foreground, border, icon, state), when styled, then the color must resolve from the Angular Material theme via Material Design system tokens or theme palette accessors (`mat.get-theme-color`, `mat.define-theme`, Material 3 system tokens such as `--mat-sys-primary`, `--mat-sys-surface`); hard-coded hex, rgb, hsl, or named CSS colors outside the theme definition file are prohibited.
6. Given the application theme, when defined, then it must be configured in a single Sass theme file using `mat.define-theme` (Material 3) with explicit primary, tertiary, and (if used) error palettes, and applied globally via `mat.all-component-themes` or `mat.all-component-colors` + typography + density mixins.
7. Given dark mode or high-contrast variants exist, when the user toggles or the system requests them, then the change is implemented by switching Angular Material theme tokens only (no parallel custom stylesheet).
8. Given any icon is rendered, when displayed, then it must use `<mat-icon>` with a Material icon font or registered SVG icon set; inline `<svg>` or `<img>` icons outside the `MatIconRegistry` are prohibited.
9. Given a form is rendered, when validated, then error, hint, and label display must use `<mat-form-field>`, `<mat-error>`, and `<mat-hint>`.
10. Given a lint/build enforcement step, when CI runs, then an automated check (stylelint rule or custom ESLint rule) fails the build if: (a) raw `<button>`, `<input>`, `<select>`, or `<textarea>` elements appear in templates without a Material directive, (b) color literals appear outside the theme file, or (c) typography properties are declared outside the theme file.
11. Given responsive breakpoints are needed, when defined, then they must use Angular CDK `BreakpointObserver` with Material's breakpoint constants rather than custom media-query values.
12. Given spacing and density, when applied, then density must be set via `mat.private-density-config` / density tokens and spacing must use a theme-derived scale, not arbitrary pixel values.

## L2-023: Account and Data Deletion
**Traces to:** L1-010, L1-009

Users must be able to delete their account and associated personal data.

**Acceptance Criteria:**
1. Given a logged-in user, when they request account deletion and confirm, then within 30 days: private notes are hard-deleted, spouse-shared notes authored by the user are hard-deleted, public reviews/comments/notes are anonymized (author replaced with "[deleted user]"), and the couple is dissolved if applicable.
2. Given a deletion request, when submitted, then a confirmation email is sent and the account is immediately disabled from login.
3. Given deletion completes, when audited, then no PII belonging to the deleted user remains in primary stores or backups older than 30 days.
